## 🔍 ภาพรวมการทดสอบ (Executive Summary)
ระบบประกอบด้วย 2 ส่วนหลักคือ **Next.js Web Application** (ระบบควบคุมฝั่งคลาวด์และจัดการสิทธิ์ของนักศึกษา) และ **ESP32 Firmware** (อุปกรณ์ฮาร์ดแวร์ควบคุมการปลดล็อกประตู) 

จากการตรวจสอบซอร์สโค้ดเชิงลึก (Static Application Security Testing - SAST) และการจำลองตรรกะการโจมตี (Threat Modeling) พบว่าสถาปัตยกรรมภาพรวมมีมาตรการความปลอดภัยที่ดีในหลายด้าน เช่น การป้องกัน SQL Injection โดยใช้ Parameterized Queries, การใช้ Bcrypt Hashing และระบบ Dynamic One-Time Token สำหรับสแกน QR Code เพื่อป้องกันการส่งต่อลิงก์

อย่างไรก็ตาม **ยังคงพบช่องโหว่ความปลอดภัยระดับรุนแรงสูง (High-Critical) 5 จุดหลัก** ทั้งบนสิทธิ์การเข้าถึงข้อมูลของนักศึกษา (IDOR/Privacy Leak) และในขั้นตอนการแลกเปลี่ยนข้อมูลสื่อสารผ่านเครือข่ายของอุปกรณ์ ESP32 IoT (MitM & Authentication Key Exfiltration) ซึ่งหากถูกโจมตีอาจส่งผลให้ประตูถูกสั่งเปิดโดยไม่มีสิทธิ์ หรือข้อมูลส่วนบุคคลของนักศึกษารั่วไหลสู่สาธารณะได้

## 🛑 รายการช่องโหว่ที่ตรวจพบและจุดที่ต้องปรับปรุง (Vulnerability Findings & Mitigation)

### 1. ช่องโหว่ IDOR / ข้อมูลส่วนบุคคลนักศึกษารั่วไหลสู่สาธารณะ (Insecure Direct Object Reference)
> **ความรุนแรง:** 🔴 สูงมาก (High - Privacy & Compliance Violation)  
> **ไฟล์ที่เกี่ยวข้อง:** [my-app/app/api/students/[id]/route.ts](file:///C:/Users/aunkh/OneDrive/Desktop/Project/my-app/app/api/students/[id]/route.ts)

*   **ลักษณะของช่องโหว่:**  
    ฟังก์ชัน `GET` ใน API `/api/students/[id]` เปิดเป็นสาธารณะ (Public Endpoint) เพื่อให้นักศึกษาสามารถดึงสถานะคำขอลงทะเบียนของตัวเองได้ แต่ระบบใช้เลข ID ลำดับฐานข้อมูล (`parseInt(id)` เป็น Serial Integer) โดยไม่มีมาตรการปกป้องสิทธิ์ใดๆ
*   **ช่องทางการโจมตี (Attack Vector):**  
    ผู้ไม่หวังดีสามารถเขียนสคริปต์สั้นๆ ไล่ลำดับตัวเลข (ID Enumeration / Scraping) ตั้งแต่ 1 ไปเรื่อยๆ (เช่น `/api/students/1`, `/api/students/2`...) เพื่อดึงรายชื่อ ชื่อจริง-นามสกุล, รหัสนักศึกษา, คณะ, สาขา และประวัติเวลาการสแกนเข้าห้องของนักศึกษาทุกคนในระบบออกมาทั้งหมด ซึ่งขัดต่อกฎหมายคุ้มครองข้อมูลส่วนบุคคล (PDPA/GDPR) อย่างรุนแรง
*   **แนวทางแก้ไข:**  
    บังคับให้ผู้ส่งคำขอต้องระบุ `bypass_token` (คีย์สุ่มความปลอดภัยสูง 24 ไบต์ที่สร้างขึ้นตอนลงทะเบียนสำเร็จ) แนบมาทาง Request Header หรือ Query Parameter เสมอ และตรวจสอบในฐานข้อมูลว่า Token นั้นตรงกับ ID ของข้อมูลนักศึกษาคนดังกล่าวจริงหรือไม่ก่อนแสดงผลข้อมูล หากไม่ระบุหรือไม่ตรงให้ปฏิเสธการดึงข้อมูล

### 2. ช่องโหว่การข้ามการตรวจสอบใบรับรองความปลอดภัยของ ESP32 (TLS/SSL MitM Vulnerability)
> **ความรุนแรง:** 🔴 สูงมาก (High - IoT Control Hijack)  
> **ไฟล์ที่เกี่ยวข้อง:** [esp32/esp32.ino](file:///C:/Users/aunkh/OneDrive/Desktop/Project/esp32/esp32.ino#L378)

*   **ลักษณะของช่องโหว่:**  
    ในเฟิร์มแวร์ ESP32 มีการตั้งค่าข้ามการยืนยันความถูกต้องของ SSL Certificate ผ่านทางคำสั่ง:
    ```cpp
    client->setInsecure(); // ข้าม CA Cert เพื่อรับความรวดเร็วในการติดต่อเซิร์ฟเวอร์
    ```
*   **ช่องทางการโจมตี (Attack Vector):**  
    หากนักศึกษาหรือผู้บุกรุกเข้าสู่วงเครือข่าย Wi-Fi เดียวกันกับบอร์ด (เช่น เครือข่าย Wi-Fi สถาบัน) เขาสามารถโจมตีแบบ **Man-in-the-Middle (MitM)** โดยการปลอมแปลง DNS หรือทำ ARP Spoofing เพื่อเบนเส้นทางการสื่อสารของ ESP32 มายังเครื่องเซิร์ฟเวอร์จำลองของแฮกเกอร์ และเนื่องจาก ESP32 ไม่ตรวจสอบใบรับรองความปลอดภัย แฮกเกอร์จึงสามารถส่งผลลัพธ์ข้อมูลปลอมเป็น JSON ที่มีค่า `"door_trigger": "open"` ส่งกลับไปให้บอร์ด เพื่อสั่งให้ประตูเปิดออกทันทีได้โดยไม่ต้องอาศัยการอนุมัติใดๆ จากแอดมินจริง
*   **แนวทางแก้ไข:**  
    เปลี่ยนจากการใช้ `.setInsecure()` มาเป็นการแนบ Root CA Certificate ของผู้ให้บริการคลาวด์/โดเมน (เช่น Vercel Root CA) ลงไปในเฟิร์มแวร์ ESP32 และตั้งค่าเปิดใช้งานตรวจสอบ SSL ให้ถูกต้อง

### 3. การเก็บคีย์ลับไว้ในเฟิร์มแวร์และค่าเริ่มต้นไม่ปลอดภัย (Hardcoded Credentials)
> **ความรุนแรง:** 🟡 ปานกลาง-สูง (Medium-High - Credential Exposure)  
> **ไฟล์ที่เกี่ยวข้อง:** [esp32/esp32.ino](file:///C:/Users/aunkh/OneDrive/Desktop/Project/esp32/esp32.ino#L26), [my-app/lib/esp32.ts](file:///C:/Users/aunkh/OneDrive/Desktop/Project/my-app/lib/esp32.ts#L20)

*   **ลักษณะของช่องโหว่:**  
    มีการระบุ `ESP32_API_KEY` ค่าเริ่มต้นไว้แบบคงที่ในซอร์สโค้ด (Hardcoded String) เป็น `"smartaccess_secure_door_unlock_token_2026"` ทั้งฝั่งบอร์ดและเซิร์ฟเวอร์ Next.js และไม่มีคำเตือนหรือหยุดการทำงานหากใช้งานค่าดีฟอลต์บนสภาพแวดล้อมใช้งานจริง (Production)
*   **ช่องทางการโจมตี (Attack Vector):**  
    1. ผู้โจมตีสามารถดึงไฟล์ข้อมูลไบนารีที่คอมไพล์แล้วออกจากชิปหน่วยความจำ Flash ของ ESP32 ได้อย่างง่ายดายทางสายสัญญาณ หรืออ่านซอร์สโค้ดจาก Git Repository หากตั้งค่าเป็น Public เพื่อดึงรหัส API Key ออกมานำไปใช้เรียกดูข้อมูล `active_token` บน Next.js API หลังบ้าน
    2. หากใช้งานคีย์เริ่มต้นนี้ในเซิร์ฟเวอร์ Production สุ่มเสี่ยงที่จะทำให้คนนอกเดาคีย์ได้ง่ายๆ
*   **แนวทางแก้ไข:**  
    1. ฝั่ง Next.js: เพิ่มระบบป้องกันใน [lib/esp32.ts](file:///C:/Users/aunkh/OneDrive/Desktop/Project/my-app/lib/esp32.ts) โดยให้แสดงแจ้งเตือนหรือหยุดการรัน (Throw Error) หากใช้งานค่าคีย์เริ่มต้นบนระบบ Production (เหมือนระบบตรวจสอบ JWT_SECRET)
    2. ฝั่ง ESP32: นำระบบจัดเก็บข้อมูลการตั้งค่าที่ไม่สูญหาย (Non-Volatile Storage - NVS/EEPROM) มาใช้เพื่อเก็บตัวแปรแทนการใส่คีย์ตายตัวลงไปในเฟิร์มแวร์โดยตรง

### 4. ไม่มีระบบป้องกันการเดาสุ่มรหัสผ่าน (Brute-Force & Credential Stuffing)
> **ความรุนแรง:** 🟡 ปานกลาง (Medium)  
> **ไฟล์ที่เกี่ยวข้อง:** [my-app/app/api/auth/login/route.ts](file:///C:/Users/aunkh/OneDrive/Desktop/Project/my-app/app/api/auth/login/route.ts)

*   **ลักษณะของช่องโหว่:**  
    ระบบยังไม่มีกลไกการหน่วงเวลาหรือจำกัดจำนวนครั้งในการพยายามล็อกอิน (Rate Limiting) ที่ API สำหรับเข้าสู่ระบบผู้ดูแลระบบ
*   **ช่องทางการโจมตี (Attack Vector):**  
    เนื่องจากค่าเริ่มต้นของบัญชีแอดมินถูกกำหนดมาให้เป็น `admin` / `admin123` ทำให้ผู้โจมตีสามารถสุ่มป้อนข้อมูลด้วยบอทความเร็วสูงเพื่อแกะรอยเดารหัสผ่านแอดมินบัญชีอื่นๆ ได้โดยระบบไม่มีการบล็อกการทำงานชั่วคราว
*   **แนวทางแก้ไข:**  
    ใช้กลไก Rate Limiter (เช่น การใช้ In-memory Map แบบจำกัดจำนวนครั้งต่อหมายเลข IP หรือการนำ middleware อัปเกรดมาใช้งาน) บน API สำหรับการลงชื่อเข้าใช้งาน

### 5. ความไม่สอดคล้องของการยืนยันสิทธิ์ในการทดสอบ Discord Webhook
> **ความรุนแรง:** 🟢 ต่ำ (Low - Privilege Inconsistency / Minor SSRF)  
> **ไฟล์ที่เกี่ยวข้อง:** [my-app/app/api/system/test-webhook/route.ts](file:///C:/Users/aunkh/OneDrive/Desktop/Project/my-app/app/api/system/test-webhook/route.ts#L8)

*   **ลักษณะของช่องโหว่:**  
    API การตั้งค่าระบบอื่นๆ ทั้งหมดมีการป้องกันระดับสูงโดยอนุญาตเฉพาะบัญชีบทบาท `owner` (แอดมินระดับสูงสุด) เท่านั้น แต่ API `/api/system/test-webhook` กลับยินยอมให้ผู้ที่ลงชื่อเข้าใช้งานที่มีบทบาท `door_operator` (เจ้าหน้าที่ควบคุมประตูทั่วไป) สามารถเรียกใช้งานเพื่อส่งคำขอ HTTP POST ไปยัง Discord Webhook ใดๆ ก็ได้
*   **แนวทางแก้ไข:**  
    เปลี่ยนเงื่อนไขการอนุมัติให้เข้มงวดสม่ำเสมอกัน โดยกำหนดสิทธิ์ `admin.role === "owner"` เสมอใน API เพื่อรักษาความปลอดภัยเชิงสิทธิ์ขั้นต่ำ (Principle of Least Privilege)
