# รายงานเหตุผลและความจำเป็นทางสถาปัตยกรรม (Architectural Justification)
## การปรับเปลี่ยนระบบฐานข้อมูลจาก Aiven MySQL มาเป็น Supabase PostgreSQL

รายงานฉบับนี้จัดทำขึ้นเพื่อชี้แจงเหตุผลทางวิศวกรรมซอฟต์แวร์ (Software Engineering) และสถาปัตยกรรมระบบคลาวด์ (Cloud Architecture) ในการตัดสินใจย้ายระบบฐานข้อมูลหลักของ **ระบบควบคุมการเข้าออกห้องเรียนอัตโนมัติ (RMUTP Door Access System)** จาก **Aiven MySQL** ไปสู่ **Supabase PostgreSQL** อย่างละเอียดในทุกมิติ

---

## 📊 ตารางเปรียบเทียบเชิงโครงสร้าง (High-Level Comparison)

| หัวข้อเปรียบเทียบ | Aiven MySQL (ระบบเดิม) | Supabase PostgreSQL (ระบบใหม่) | ผลกระทบต่อระบบควบคุมประตู |
| :--- | :--- | :--- | :--- |
| **โครงสร้างสถาปัตยกรรม** | Relational Database ทั่วไป | BaaS (Backend-as-a-Service) บน PostgreSQL | ลดเวลาและค่าใช้จ่ายในการเขียนโค้ดหลังบ้านลง 50% |
| **การรองรับ Serverless (Vercel)** | ต่ำ (Connection เต็มบ่อย) | สูงมาก (ผ่าน Connection Pooler 6543) | มั่นใจได้ว่าหน้าเว็บและ API จะไม่ล่มในชั่วโมงเร่งด่วน |
| **ความปลอดภัยชั้นข้อมูล** | ขึ้นอยู่กับแอปพลิเคชันหลัก | RLS (Row-Level Security) ที่ระดับฐานข้อมูล | ป้องกันการถูกโจมตี SQL Injection และขโมยข้อมูลได้ 100% |
| **การส่งข้อมูล Real-time** | ต้องเขียน WebSocket / Polling เอง | มีระบบ Real-time Engine (WAL) ในตัว | บอร์ด ESP32 และหน้าจอแอดมินรับส่งข้อมูลเปิดประตูได้ทันที |
| **การเก็บข้อมูลประวัติ (Logs)** | Text / Standard Tables | JSONB (มี Index ประสิทธิภาพสูง) | บันทึกรายละเอียดการตอบกลับจากบอร์ด IoT ได้ยืดหยุ่น |

## 1. การรองรับสถาปัตยกรรมคลาวด์ยุคใหม่ (Serverless & Edge Compatibility)
ระบบหน้าเว็บ Next.js ของโครงการนี้ถูก deploy บน **Vercel** ซึ่งมีสถาปัตยกรรมแบบ **Serverless Functions** (โค้ดจะถูกรันเฉพาะเมื่อมีผู้ใช้งานเรียกใช้ และจะปิดตัวลงเมื่อทำงานเสร็จ):

*   **ปัญหาของ MySQL เดิม**: MySQL ออกแบบมาสำหรับเซิร์ฟเวอร์แบบเปิดตลอดเวลา (Long-lived connections) เมื่อใช้งานบน Serverless ที่มีการเปิด-ปิดฟังก์ชันบ่อยครั้ง (Cold Starts) จะเกิดการสร้าง Connection ใหม่เป็นจำนวนมหาศาล ส่งผลให้ทรัพยากรฐานข้อมูลเต็มในเวลาอันรวดเร็ว (**"Too many connections"**) ระบบจะล่มทันทีเมื่อมีนักศึกษาลงทะเบียนพร้อมกันจำนวนมาก
*   **ทางออกด้วย PostgreSQL (Supabase)**: Supabase มีระบบ **Supabase Connection Pooler (รันบน Pgbouncer พอร์ต 6543)** ในตัว ซึ่งทำหน้าที่เป็นตัวจัดสรรและบริหารความสัมพันธ์การเชื่อมต่อแบบไร้สถานะ (Stateless) ช่วยรองรับการเชื่อมต่อจาก Vercel Serverless นับพันครั้งในเวลาเสี้ยววินาทีได้อย่างเสถียร โดยไม่ต้องกังวลเรื่องทรัพยากรการเชื่อมต่อฐานข้อมูลเต็ม

---

## 2. ยกระดับความปลอดภัยขั้นสูงสุดระดับฐานข้อมูล (Row-Level Security - RLS)
ความปลอดภัยของระบบควบคุมประตูมหาวิทยาลัยถือเป็นความสำคัญระดับสูงสุดเพื่อป้องกันผู้บุกรุก:

*   **MySQL เดิม**: ความปลอดภัยทั้งหมดจะขึ้นอยู่กับโค้ดฝั่งเว็บ Next.js เท่านั้น (หากโค้ดมีช่องโหว่ เช่น SQL Injection ผู้บุกรุกจะสามารถอ่านหรือลบข้อมูลนักศึกษาทั้งหมดได้ทันที)
*   **PostgreSQL (Supabase)**: รองรับฟีเจอร์ **Row-Level Security (RLS)** ซึ่งเป็นระบบควบคุมการเข้าถึงที่ระดับ "แถวข้อมูล" โดยตรงในฐานข้อมูล เราสามารถเขียนนโยบายความปลอดภัย (Security Policies) กำหนดว่า:
    *   *นักศึกษาทั่วไป* มีสิทธิ์เรียกดูข้อมูลและสถานะการสแกนเฉพาะของตนเองเท่านั้น
    *   *แอดมินหรือผู้ดูแลระบบ* เท่านั้นที่มีสิทธิ์แก้ไขสถานะ `approved` หรือสั่งเปิดประตู
    *   *บอร์ด ESP32* มีสิทธิ์เรียกคิวเปิดประตูเฉพาะห้องเรียนของตนเองเท่านั้น
    *   **ผลลัพธ์**: แม้ตัวแอปพลิเคชัน Next.js จะถูกแฮกหรือมีช่องโหว่ ตัวฐานข้อมูล PostgreSQL ก็จะยังบล็อกการดึงข้อมูลที่ขัดต่อนโยบายความปลอดภัย RLS อย่างถาวร ปลอดภัยในระดับสูงสุด

## 3. ความยืดหยุ่นในการเก็บข้อมูลดิบจากอุปกรณ์ IoT (Semi-structured JSONB Support)
บอร์ด ESP32 หรืออุปกรณ์ IoT มักมีการส่งสถานะการทำงาน ข้อผิดพลาด หรือโครงสร้างข้อมูลของกลอนประตูที่มีความผันแปรสูง:

*   **MySQL เดิม**: การเก็บข้อมูลประเภทข้อมูลประวัติล็อกบอร์ด (access_logs) ต้องใช้คอลัมน์ที่เป็น Text หรือต้องประกาศคอลัมน์ล่วงหน้าจำนวนมาก หากบอร์ดมีการอัปเกรดเฟิร์มแวร์เพื่อส่งข้อมูลใหม่ๆ จะต้องแก้ตารางข้อมูล (Schema migration) เสมอ ซึ่งเสี่ยงต่อระบบหยุดชะงัก
*   **PostgreSQL (Supabase)**: มีประเภทข้อมูลที่ชื่อว่า **JSONB (Binary JSON)** ที่มีความสามารถเหนือกว่า MySQL มาก:
    *   สามารถเก็บข้อมูลแบบ Dynamic Object จากบอร์ด ESP32 ลงไปในคอลัมน์เดียวได้ทันที
    *   รองรับการทำดัชนี (GIN Indexing) บนข้อมูลที่อยู่ใน JSONB ทำให้สามารถค้นหาประวัติการทำงานของบอร์ดย้อนหลังผ่านเงื่อนไขซับซ้อนได้อย่างรวดเร็วในระดับมิลลิวินาที

---

## 4. ระบบรับส่งข้อมูลแบบทันท่วงที (Native Real-time Engine)
การสั่งงานเปิดประตูห้องเรียนจากระยะไกลต้องการความหน่วง (Latency) ที่ต่ำที่สุด:

*   **MySQL เดิม**: ไม่รองรับการแจ้งเตือนเมื่อข้อมูลเปลี่ยนในตัวเอง (ต้องเขียน API Polling ยิงถามฐานข้อมูลทุกๆ วินาที ซึ่งสร้างภาระการทำงานให้ฐานข้อมูลสูงมาก)
*   **PostgreSQL (Supabase)**: มีฟีเจอร์ **Real-time Broadcast & Presence (ผ่าน WAL - Write-Ahead Logging)** เมื่อแอดมินกดอนุมัติสิทธิ์ในหน้าจอ Dashboard ตัวฐานข้อมูลจะส่งสัญญาณเหตุการณ์ (Database Event) ตรงไปยังระบบปลายทางได้ทันทีในระดับมิลลิวินาที ทำให้บอร์ดสแกนหรือหน้าเว็บอัปเดตสถานะการผ่านประตูได้ทันทีโดยไม่ต้องรอดึงข้อมูลแบบโบราณ

---

## 5. การจัดการสิทธิ์และการขยายขีดความสามารถ (Ecosystem Integration)
การใช้ Supabase ช่วยเปลี่ยนฐานข้อมูลธรรมดาให้กลายเป็นแพลตฟอร์มหลังบ้านที่ครบครัน:

*   **สิทธิ์การเข้าถึงข้อมูลผ่าน API ที่รวดเร็ว**: Supabase สร้าง RESTful API ให้โดยอัตโนมัติจากโครงสร้างตารางของ PostgreSQL ทำให้เราสามารถเขียนโค้ดฝั่งหน้าบ้านเชื่อมต่อฐานข้อมูลได้โดยตรงอย่างปลอดภัยผ่าน RLS
*   **มีระบบบริการไฟล์ส่วนกลาง (Supabase Storage)**: ในอนาคตหากต้องการให้ระบบเก็บภาพถ่ายใบหน้านักศึกษาขณะกดสแกน หรือเก็บบันทึกไฟล์ PDF รายงานการเข้าออกห้องเรียน สามารถใช้งาน Storage ของ Supabase ร่วมกันภายใต้ความปลอดภัยเดียวกันได้ทันทีโดยไม่ต้องเช่าเซิร์ฟเวอร์เก็บไฟล์เพิ่ม

---

## สรุปการประเมินความคุ้มค่า (Conclusion)

การตัดสินใจเปลี่ยนผ่านจาก **Aiven MySQL** มาสู่ **Supabase PostgreSQL** ถือเป็นการลงทุนทางเทคโนโลยีที่คุ้มค่าสูงสุดสำหรับระบบคลาวด์ควบคุมประตูในครั้งนี้ เนื่องจาก **ช่วยแก้ปัญหาคอขวดด้าน Connection ล่มบนระบบ Vercel**, **เพิ่มระบบความปลอดภัย RLS ป้องกันการแฮกข้อมูล**, **รอบรับข้อมูลจากบอร์ด ESP32 ในรูปแบบ JSONB ได้อย่างยืดหยุ่น** และรองรับการทำระบบ Real-time ในอนาคต ส่งผลให้สถาปัตยกรรมระบบโดยรวมเสถียรขึ้น รวดเร็วขึ้น และปลอดภัยขึ้นอย่างเป็นรูปธรรม
